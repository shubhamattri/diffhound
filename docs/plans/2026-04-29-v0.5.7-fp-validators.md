## Fix A.1: `lib/validators/migration-column-check.sh` (new)

**What it does:** When a FINDING's WHAT names a migration filename (`\d{14}_[a-z_]+\.ts`) and claims a column is missing, grep the named migration file. If the column literal exists, drop the finding.

**Pseudocode:**
```
read FINDING block on stdin
extract WHAT
m_name ‚Üê regex \d{14}_[a-z_]+\.ts (first match)        # e.g. 20260402000000_br_deck_jobs.ts
columns ‚Üê snake_case identifiers near tokens
         "no `‚Ä¶` column", "missing `‚Ä¶` column",
         "no `‚Ä¶` or `‚Ä¶` columns"
[ -z m_name ] && pass through
for col in columns:
  if grep -qE "[\"']${col}[\"']" $REPO/services/api/migrations/$m_name:
    DROP finding, annotate "[migration-column-check: column '<col>' exists in '<m_name>' ‚Äî claim contradicted]"
    break
```

**Repo path resolution:** uses `$REVIEW_REPO_PATH` (set by `bin/diffhound`) plus the conventional `services/api/migrations/` path. Falls back to `.` if not set. If migration file isn't found on disk, no-op (we can only drop on positive evidence).

**False-drop risk:** Near-zero. Only drops when the literal column string appears in the named migration.

**Pipeline order:** Runs after `citation-discipline.sh`, before `dedup-helper.py`. Matches the existing `lib/validators/run-all.sh` order.

## Fix B: extend `lib/validators/citation-discipline.sh`

**What it does:** Detect prose claiming an unverifiable tool run; downgrade severity by one tier.

**Patterns (case-insensitive):**
```
i (ran|grep'd|searched|checked) .* (got|found|saw|returned) (zero|no|0) (hits|matches|results)
confirmed with (grep|search|find)
verified .* (zero|no|0) (hits|matches)
ran a grep .* across .* and got
```

**Action:** BLOCKING ‚Üí SHOULD-FIX ‚Üí NIT ‚Üí drop. Annotate `[citation-discipline: unverifiable tool-run claim ‚Äî downgraded from <X>]`.

**Why downgrade rather than drop:** prose like this could coincidentally appear in a true finding. Downgrade ensures it doesn't anchor a review at BLOCKING based on hallucinated evidence; if the underlying claim is real, it surfaces in next review at lower noise.

**Stacks with A.1:** if both A.1 and B fire on F1/F2, the finding is dropped (A.1) regardless of severity (B's downgrade) ‚Äî defense in depth.

## Fix C: extend `lib/validators/ref-exists.sh`

**Current behavior:** verifies symbol existence in REACHABLE_PATH for backticked types/classes.

**Extension:** also verify backticked function names cited in WHAT. Search REACHABLE_PATH file + same-directory siblings for `function ${sym}` / `const ${sym}` / `${sym}\s*=` / `import.*${sym}`. If zero hits across all candidates, drop with `[ref-exists: function '<sym>' not found in cited paths or same dir]`.

**Skiplist:** built-ins, jest globals (`expect`, `it`, `describe`), node stdlib names ‚Äî reuse the existing skiplist. Don't expand it speculatively.

## Fix D: `lib/validators/no-validation-check.sh` (new)

**Trigger:** WHAT contains `no\s+(format\s+)?validation`, `no\s+\w+\s+check`, `missing\s+validation`, `no\s+format\s+check` ‚Äî AND names a specific backticked function.

**Action:**
```
locate function definition in REACHABLE_PATH (regex: function FN\b|const FN\s*=|export.*function FN\b)
read function body up to matching close brace (rough ‚Äî until next top-level export or ^})
count validation tells in body:
  RegExp.test, \.test\(, if\s+\(\s*!, throw new, validate, assert,
  instanceof, typeof\s+\w+\s+===, Number\.isFinite, Array\.isArray,
  schema\.parse, \.safeParse
if tells_count >= 2:
  DROP, annotate "[no-validation-check: function '<fn>' has <N> validation calls; claim contradicted]"
```

**False-drop risk:** Low. A function with no validation has zero tells. Two-tells threshold guards against incidental matches in unrelated code.

**Brace-matching:** rough ‚Äî use awk to find first matching `^}` after the function definition line. Imperfect on heavily nested closures but adequate for the "did the LLM read the body at all" question.
